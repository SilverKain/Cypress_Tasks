# Anki Cards: Week 7-8 - Bug Lifecycle & Documentation
# Карточки для импорта в Anki: Жизненный цикл дефекта, статусы, приоритеты, документация
# Формат: Фронт карточки [TAB] Бэк карточки
# Количество карточек: 200

Critical Bug	Критерии: Блокирует критичную функциональность, препятствует дальнейшему тестированию, угрожает безопасности данных, вызывает потерю данных. Примеры: Невозможность авторизации, падение системы при запуске, SQL-инъекции, потеря платежных данных. Действия QA: Немедленная эскалация, детальное воспроизведение, приоритет в исправлении, обязательная верификация.

Major Bug	Критерии: Существенно влияет на функциональность, но есть workaround, влияет на бизнес-процессы, снижает производительность. Примеры: Неработающий поиск, некорректные расчеты, неправильное отображение данных. Действия QA: Документирование workaround, планирование исправления в текущем релизе, информирование stakeholders.

Minor Bug	Критерии: Косметические дефекты, незначительные отклонения от требований, не влияет на основную функциональность. Примеры: Опечатки в тексте, неправильные цвета, мелкие UI несоответствия. Действия QA: Группировка в batch для исправления, включение в backlog, проверка в regression testing.

Trivial Bug	Критерии: Минимальное влияние на пользователя, предложения по улучшению, enhancement requests. Примеры: Предложения по UI, оптимизация текстов, улучшение usability. Действия QA: Документирование как improvement, обсуждение приоритета с Product Owner, включение в будущие релизы.

High Priority	Критерии: Требует срочного исправления, блокирует релиз, влияет на ключевые сценарии, связан с deadline. Примеры: Bugs в production, критичные бизнес-функции, security уязвимости. Действия QA: Ежедневное отслеживание, координация с разработкой, подготовка к hotfix, немедленное тестирование исправлений.

Medium Priority	Критерии: Должно быть исправлено в текущем релизе, есть workaround, умеренное влияние на пользователей. Примеры: Функциональные ошибки с обходными путями, performance issues, интеграционные проблемы. Действия QA: Планирование в спринт, регулярный мониторинг статуса, подготовка тест-кейсов для верификации.

Low Priority	Критерии: Может быть отложено, минимальное влияние, косметические изменения. Примеры: UI improvements, minor text changes, optimization suggestions. Действия QA: Включение в backlog, группировка с подобными issues, тестирование при наличии времени.

Bug Life Cycle	Критерии: Полный цикл от обнаружения до закрытия: New → Open → Assigned → In Progress → Fixed → Retest → Verified → Closed. Примеры: Defect workflow в Jira, статусы в TestRail, transitions в Azure DevOps. Действия QA: Контроль переходов статусов, участие в triage, верификация исправлений, closure validation.

New Status	Критерии: Дефект только что создан, требует первичного анализа, ожидает triage, не назначен исполнителю. Примеры: Bug только что найден тестировщиком, автоматически созданный из мониторинга, пользовательский репорт. Действия QA: Первичная верификация, добавление деталей, присвоение severity/priority, подготовка к triage.

Open Status	Критерии: Дефект подтвержден, прошел triage, принят к исправлению, находится в backlog команды. Примеры: Validated bug после team review, confirmed defect в Sprint Planning, approved issue для разработки. Действия QA: Мониторинг assignment, уточнение требований при необходимости, подготовка test data для fix verification.

Assigned Status	Критерии: Дефект назначен конкретному разработчику, есть ответственный за исправление, включен в план работ. Примеры: Developer взял bug в работу, task assigned в Sprint, ownership установлен. Действия QA: Координация с назначенным разработчиком, предоставление дополнительной информации, мониторинг progress.

In Progress Status	Критерии: Разработчик активно работает над исправлением, ведется coding/debugging, progress visible. Примеры: Code changes в процессе, debugging session активен, developer updates status. Действия QA: Поддержка debugging процесса, предоставление test environment, готовность к code review.

Fixed Status	Критерии: Разработчик завершил исправление, код готов к тестированию, fix deployed в test environment, требует verification. Примеры: Code merged в feature branch, build готов для testing, developer confirms fix completion. Действия QA: Немедленное планирование retest, подготовка verification scenarios, координация test environment access.

Retest Status	Критерии: QA активно тестирует исправление, выполняется verification process, проводится regression testing. Примеры: Execution of verification test cases, regression testing в affected areas, validation of fix effectiveness. Действия QA: Тщательная проверка fix, regression testing, documentation результатов, communication с разработчиком при необходимости.

Verified Status	Критерии: QA подтвердил исправление дефекта, fix работает корректно, regression testing passed, ready for closure. Примеры: Test cases passed, no side effects found, functionality restored, stakeholder approval received. Действия QA: Финальная документация результатов, communication stakeholders, подготовка к closure, обновление test cases при необходимости.

Closed Status	Критерии: Дефект полностью исправлен и проверен, все stakeholders согласны с closure, fix deployed в production. Примеры: Successful production deployment, user acceptance confirmed, no reopening requests. Действия QA: Final status update, lessons learned documentation, metrics update, closure confirmation в reporting.

Reopened Status	Критерии: Ранее закрытый дефект возник снова, fix оказался неполным, новые проявления старой проблемы. Примеры: Bug recurrence после deployment, incomplete fix discovery, regression в новой версии. Действия QA: Повторный анализ root cause, обновление bug description, escalation к original developer, additional test scenarios creation.

Deferred Status	Критерии: Дефект отложен для будущих релизов, низкий приоритет, resource constraints, business decision. Примеры: Minor UI issues, enhancement requests, performance optimizations, low-impact bugs. Действия QA: Justification documentation, backlog inclusion, periodic review scheduling, impact reassessment planning.

Rejected Status	Критерии: Дефект признан invalid, not reproducible, by design, duplicate или not a bug. Примеры: User misunderstanding, intended behavior, environment-specific issues, duplicate reports. Действия QA: Подробное обоснование rejection, documentation для future reference, communication с reporter, knowledge base update.

Duplicate Status	Критерии: Идентичный дефект уже существует в системе, refers к существующему ticket, multiple reports одной проблемы. Примеры: Same issue reported by different testers, multiple user reports, related bugs consolidated. Действия QA: Link к original ticket, consolidation information, notification всех reporters, tracking centralization.

Won't Fix Status	Критерии: Дефект не будет исправляться по business reasons, architectural limitations, cost considerations. Примеры: Legacy code limitations, third-party dependencies, design decisions, cost-benefit analysis. Действия QA: Business justification documentation, workaround provision, user communication, knowledge base update.

Invalid Status	Критерии: Репорт не является дефектом, incorrect understanding, configuration issue, user error. Примеры: Feature working as designed, user training issue, environment misconfiguration, documentation gap. Действия QA: Clear explanation provision, documentation reference, user education, FAQ update.

Cannot Reproduce	Критерии: Unable to reproduce заявленную проблему, insufficient information, environment-specific issue, intermittent bug. Примеры: Rare timing issues, specific configuration dependencies, unclear reproduction steps, missing test data. Действия QA: Additional information request, different environment testing, collaboration с reporter, detailed investigation.

Need More Info	Критерии: Недостаточно информации для анализа, требуются дополнительные детали, unclear reproduction steps. Примеры: Vague bug descriptions, missing screenshots, incomplete logs, unclear expected behavior. Действия QA: Specific information requests, template provision, collaboration с reporter, clarification meetings.

Bug Report Template	Критерии: Стандартизированный формат для описания дефектов: Title, Environment, Steps to Reproduce, Expected Result, Actual Result, Severity, Priority, Attachments. Примеры: Jira bug template, TestRail defect form, Azure DevOps work item template. Действия QA: Template enforcement, quality validation, completeness verification, standardization maintenance.

Reproduction Steps	Критерии: Пошаговая инструкция для воспроизведения дефекта, должны быть clear, detailed, и repeatable. Примеры: "1. Open login page 2. Enter invalid credentials 3. Click Login button 4. Observe error message". Действия QA: Clarity verification, step-by-step validation, different tester reproduction, simplification при возможности.

Expected vs Actual Result	Критерии: Четкое описание ожидаемого поведения vs фактического результата, основанное на requirements. Примеры: Expected: "Success message displayed", Actual: "Error 500 returned". Действия QA: Requirements reference, clear description writing, screenshot provision, behavioral comparison documentation.

Environment Information	Критерии: Полная информация о test environment: OS, browser version, application version, device details, network conditions. Примеры: "Windows 10, Chrome 118, App v2.3.1, iPhone 15, WiFi connection". Действия QA: Complete environment documentation, version tracking, configuration details, reproducibility across environments.

Severity Assessment	Критерии: Оценка технического влияния дефекта на систему: Critical, High, Medium, Low. Примеры: Critical - система не запускается, High - основная функция не работает, Medium - minor functionality affected, Low - cosmetic issues. Действия QA: Technical impact analysis, system stability evaluation, functionality assessment, business impact consideration.

Priority Assignment	Критерии: Бизнес-приоритет исправления дефекта: Urgent, High, Normal, Low. Примеры: Urgent - production outage, High - affects key users, Normal - standard workflow, Low - future consideration. Действия QA: Business impact analysis, stakeholder consultation, deadline consideration, resource availability assessment.

Defect Classification	Критерии: Категоризация дефектов по типам: Functional, Performance, UI/UX, Security, Integration, Data. Примеры: Functional - feature не работает, Performance - slow response, UI - layout issues, Security - data exposure. Действия QA: Accurate categorization, impact area identification, specialist assignment, targeted testing approach.

Root Cause Analysis	Критерии: Исследование fundamental причины дефекта для prevention: Code defect, Design flaw, Requirements gap, Environment issue. Примеры: Code logic error, missing requirement, integration misconfiguration, third-party dependency. Действия QA: Analysis facilitation, pattern identification, prevention recommendations, process improvement suggestions.

Defect Metrics	Критерии: Количественные показатели для tracking defect trends: Defect density, discovery rate, fix rate, escape rate. Примеры: 5 defects per 1000 LOC, 10 bugs/day discovery, 95% fix rate, 2% escape rate. Действия QA: Metrics collection, trend analysis, reporting stakeholders, improvement opportunities identification.

Defect Triage Process	Критерии: Процесс оценки и приоритизации дефектов командой: Review, categorization, assignment, scheduling. Примеры: Daily triage meetings, severity/priority matrix usage, resource allocation decisions. Действия QA: Triage meeting participation, defect presentation, priority discussion, consensus facilitation.

Bug Tracking Tool	Критерии: Система для управления defect lifecycle: creation, tracking, reporting, analysis capabilities. Примеры: Jira, Azure DevOps, TestRail, Bugzilla, Mantis. Действия QA: Tool administration, workflow configuration, reporting setup, team training, best practices enforcement.

Defect Prevention	Критерии: Proactive меры для предотвращения дефектов: Process improvement, training, reviews, automation. Примеры: Code reviews, requirements analysis, design reviews, test automation, static analysis. Действия QA: Prevention strategy development, process enhancement, team collaboration, quality gates implementation.

Regression Testing for Fixes	Критерии: Тестирование для проверки что fix не нарушил existing functionality: Impact analysis, test selection, execution. Примеры: Related functionality testing, integration points validation, user scenarios verification. Действия QA: Regression scope definition, test case selection, automation utilization, comprehensive validation.

Defect Verification Process	Критерии: Процесс подтверждения исправления дефекта: Test case execution, scenario validation, acceptance confirmation. Примеры: Original scenario retest, edge cases validation, integration testing, user acceptance testing. Действия QA: Verification planning, thorough testing, result documentation, stakeholder communication.

Fix Validation Criteria	Критерии: Критерии successful defect resolution: Original issue resolved, no side effects, requirements met, stakeholder approval. Примеры: Feature works as expected, no new bugs introduced, performance maintained, user acceptance achieved. Действия QA: Criteria establishment, validation checklist, comprehensive testing, sign-off procedures.

Defect Closure Criteria	Критерии: Условия для closing defect: Fix verified, testing complete, stakeholder approval, production deployment. Примеры: All test cases passed, no regression found, business approval received, live environment validated. Действия QA: Closure checklist verification, final validation, documentation completion, stakeholder confirmation.

Hotfix Process	Критерии: Expedited процесс для critical defect fixes в production: Emergency assessment, rapid development, fast-track testing. Примеры: Production outage fix, security vulnerability patch, data corruption repair. Действия QA: Emergency testing protocols, risk assessment, rapid validation procedures, expedited deployment support.

Defect Documentation Standards	Критерии: Consistent стандарты для defect documentation: Template compliance, detail completeness, attachment inclusion, follow-up tracking. Defect Aging Analysis	Критерии: Анализ времени жизни дефектов для process improvement: Age tracking, bottleneck identification, resolution time analysis.
